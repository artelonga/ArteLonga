## Template repetido por type

Cada um dos 4 segue exatamente o pattern AL-1/AL-2/AL-26:

1. **Schema** já parcial em openapi/artelonga.yaml (Service, Mission, Solution) — refinar.
2. **Extract one-shot:** `tools/extract-<type>.mjs` lê data.js, gera files.
3. **Bake permanent:** `tools/bake-<type>.mjs` lê yamls + regenera bloco AUTO-GENERATED em data.js + valida.
4. **Order file:** `tools/<type>-order.txt` pra ordem canônica.
5. **validate-yaml** estende `--type=service|mission|solution|finance`.
6. **CLAUDE.md** doc instrução de "como editar".
7. **CHANGELOG entry** + V bump.

## Implementation order (dependências internas mínimas)

1. **Services** (~50 entries, maior impacto operacional) — mata L-007 se bake auto-cria shells.
2. **Missions** (~5 entries, simples).
3. **Solutions** (~5 entries, estrutura mais rica — platforms, bundledServices, lifecycle).
4. **Finances** (aggregated, não per-item — pode ficar como `recursos/finances.yaml` single file).

Cada subtarefa é commit separado dentro do mesmo PR pra rastreabilidade
(conventional commits: `refactor(AL-46-services): ...`, etc).

## Sub-tasks (commits dentro do PR)
