GIVEN AL-22 estabeleceu `openapi/artelonga.yaml` como single source of
truth pras shapes, AND `<handle>/profile.yaml` + `<handle>/community.yaml`
ainda eram editados sem validation cross-checada (só `REQUIRED` mínimo
em bake-people.mjs verifica handle/type/nome), AND um typo em campo
não detectado em commit/edit time só aparece em runtime (renderer
crash) ou nunca (bug silencioso),

WHEN adicionamos `tools/validate-yaml.mjs` que carrega openapi spec
via `js-yaml` + `ajv` (JSON Schema 2020-12 validator), valida cada
profile.yaml contra `#/components/schemas/Person` e cada
community.yaml contra `#/components/schemas/Community`, AND wire a
validation no início de `bake-people.mjs` e `bake-communities.mjs`
(pre-flight — bake aborta se schema fail antes de tocar data.js),

THEN typos/types errados/required missing em YAMLs são detectados
no momento do `npm run bake` com mensagem específica (qual file,
qual field, o que falhou), AND o openapi spec deixa de ser doc
estático pra ser contract enforced, AND `data.js` nunca contém
shapes inválidos.

---

## Estado atual (pré-AL-25)

`bake-people.mjs` tinha `REQUIRED = ['handle', 'type', 'nome']` — só
checa esses 3 campos como obrigatórios. Outros aspectos (tipo correto,
enum válido, format de date) sem validation. `bake-communities.mjs`
sequer tem REQUIRED check.

`openapi/artelonga.yaml` (criado em AL-22) descreve schemas detalhados
mas só pra documentação humana — nada faz enforcement.

Resultado: edição de YAML pode introduzir bug silencioso (ex: typo
`bioCurta` → `biocurta`, field ignorado pelo renderer) que só aparece
em prod.

## Implementação

### 1. `tools/validate-yaml.mjs`

```js
import Ajv from 'ajv/dist/2020.js';
import addFormats from 'ajv-formats';

const ajv = new Ajv({ strict: false, allErrors: true });
addFormats(ajv);

const spec = yaml.load(fs.readFileSync('openapi/artelonga.yaml'));
for (const [name, schema] of Object.entries(spec.components.schemas)) {
    ajv.addSchema(schema, `#/components/schemas/${name}`);
}

const validatePerson = ajv.compile({ $ref: "#/components/schemas/Person" });
const validateCommunity = ajv.compile({ $ref: "#/components/schemas/Community" });

// glob profile.yaml + community.yaml, validate each
```

CLI flags:
- `--type=people` — só profile.yaml
- `--type=community` — só community.yaml
- `<handle>` — só esse handle (filtro adicional)

### 2. Wire em bake-people / bake-communities

No topo de cada bake, antes de qualquer fs write:

```js
try {
    execSync('node tools/validate-yaml.mjs --type=people', {
        cwd: root, stdio: 'inherit',
    });
} catch {
    console.error('[bake-people] ABORT: schema validation failed.');
    process.exit(1);
}
```

Se validation falha, bake **não escreve** data.js — protege source of
truth de shape inválido.

### 3. Schema iteration (real data feedback)

Primeiro run revelou 12 divergências entre schema "ideal" e dados
reais:
- `Tag` enum incomplete (tinha 8, real tem 10: `aposentado`, `em-breve`,
  `menor`, `referencia`).
- `birthDate` exigia `format: date-time` mas real data tem mix de
  `date-only` e `date-time`.
- `bioCurta` tinha `maxLength: 280` mas Bruna tem 360 chars.
- `Citacao.data` exigia `format: date` mas Yuri usa `"2015"` (só ano).
- `externalUrl` exigia `format: uri` mas retro-umarizal usa
  `/retro-umarizal/` (path relativo).

Schema relaxado pra refletir realidade. Princípio: schema reflete
dados, não aspirações.

### 4. devDeps

- `ajv: ^8.17.0` — JSON Schema 2020-12 validator (compatível com OpenAPI 3.1).
- `ajv-formats: ^3.0.1` — format validators (date, date-time, uri, email).

### 5. Scripts

```json
"validate-yaml": "node tools/validate-yaml.mjs"
```

Plus integrado em ambos bakes (não exposto como step separado — corre
automático).

## Out of scope

- Validar dados em data.js post-bake. Não necessário — bake gera
  estritamente da source validada.
- Lint outras seções de data.js (services, missions). AL-N futuro
  quando essas migrarem pra YAML (AL-20+).
- CI step rodando validate-yaml em PRs. Logical next step mas
  separado.
- Schema codegen `openapi-typescript` pra `src/types.ts`. AL-22
  deixou como follow-up.
- IDE plugin pra YAML validation enquanto edita (yaml-language-server
  + schema reference). Plus, mas não é blocker.

## Notas

- AJV 8 com Ajv2020 import valida JSON Schema 2020-12 (a base do
  OpenAPI 3.1 schema dialect).
- `strict: false` desliga warnings sobre keywords não-standard que
  OpenAPI usa (`nullable`, `examples`, etc) — não muda semântica de
  validation.
- Validate roda 2× por `npm run bake` (uma por bake). Sub-segundo
  cada — não importa.
- Schema ficou intencionalmente permissivo pra Tag enum (12 valores
  que cobre todos hoje). Quando adicionar tag nova, adicionar no enum
  + commit junto. Validate força isso.
