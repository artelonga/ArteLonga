GIVEN site não passou por audit a11y formal, AND tem padrões que
podem falhar WCAG (cores cinza claro #888/#999 em fundo branco, falta
de aria-labels em chips/buttons, contrast ratios não medidos),

WHEN rodamos axe-core/Pa11y/Lighthouse audit em páginas principais,
AND corrigimos issues críticos (contrast, aria, keyboard navigation,
focus states, semantic HTML),

THEN site atende WCAG AA pra páginas principais. Audit script pode
rodar em CI pra prevenir regressão.

## Páginas-alvo

`/`, `/parceiros/`, `/yuri/`, `/servicos/grafite/`, `/contato/`, `/design/`.

## Implementação

1. `npm install --save-dev @axe-core/playwright` ou Pa11y.
2. `tests/a11y/audit.spec.ts` — roda axe em cada página, fails em
   violations críticas/sérias.
3. Coleta baseline de issues.
4. Fix em ordem: contrast > keyboard > aria > semantic.
5. CI integration (GH Action).

## Critérios de aceitação

- [ ] Baseline a11y audit documentado (issues catalogados).
- [ ] Issues críticas/sérias resolvidas.
- [ ] WCAG AA contrast em texto principal (>= 4.5:1).
- [ ] Keyboard navigation funcional (Tab/Enter/Esc).
- [ ] Focus visible em todos os inputs/buttons.
- [ ] Aria labels em chips, search input, location filters.
- [ ] CI a11y check em PRs.

## Out of scope

- WCAG AAA.
- Screen reader manual test (precisa user com VoiceOver/NVDA).
- Color blindness simulation tests (use case avaliado em design system).
