GIVEN o commit `80808b40` editou bio do Mono diretamente em
`assets/data.js` (~1700 chars de bio injetados na linha 1 do array
people), AND `data.js` é AUTO-GENERATED pelo bake (AL-1) — qualquer
edição direta é silenciosamente sobrescrita pelo próximo run, AND nada
no workflow atual previne isso (audit-shells e audit-consistency
checam OUTROS aspectos), AND essa lesson L-021 vai morder de novo se
não tivermos guard em commit time,

WHEN adicionamos `tools/pre-commit-check.mjs` que (a) snapshot o
data.js, (b) roda bake-people + bake-communities, (c) compara bytes,
(d) falha com instruções claras se houve drift, AND wire via
`.husky/pre-commit` (devDep husky), AND documentamos em CLAUDE.md
como ativar o hook,

THEN qualquer tentativa de commitar com `data.js` dessincronizado dos
YAMLs falha em commit time com mensagem útil ("você esqueceu de bake
OU editou direto no bloco AUTO-GENERATED — fix: …"). Mono incident
não pode mais acontecer silenciosamente.

## Estado atual

`tools/audit-shells.mjs` + `audit-consistency.mjs` já existem (al-16
PR) mas ambos checam questões DIFERENTES:
- audit-shells: cada slug tem shell HTML
- audit-consistency: profile.communities ↔ community.membros bidirecional

Nenhum dos dois detecta drift de data.js vs YAML source-of-truth. L-021
fica desprotegida.

`docs/LESSONS.md` já documenta L-021 com a mitigação preventiva
proposta como esse ticket: pre-commit hook bloqueando.

## Implementação

### 1. `tools/pre-commit-check.mjs`

```js
// pseudo
const before = readFileSync("assets/data.js");
exec("node tools/bake-people.mjs");
exec("node tools/bake-communities.mjs");
const after = readFileSync("assets/data.js");
if (before !== after) {
    writeFileSync("assets/data.js", before); // restore
    console.error("❌ data.js dessincronizado. Fix: npm run bake + git add + commit");
    process.exit(1);
}
```

Bake só toca blocos AUTO-GENERATED. Edições em outras seções (services,
missions, etc) sobrevivem ao bake → não disparam falha. Edições no
bloco auto OU YAMLs não sincados → disparam falha.

### 2. `.husky/pre-commit`

Wire ao git via husky (`prepare` script auto-instala em `npm install`):

```sh
#!/usr/bin/env sh
. "$(dirname "$0")/_/husky.sh"
node tools/pre-commit-check.mjs
```

### 3. `package.json`

- devDep `husky: ^9.1.0`
- script `prepare: husky`
- script `pre-commit: node tools/pre-commit-check.mjs` (manual run)

### 4. CLAUDE.md

Section "Pre-commit hook" documentando:
- Como ativar: `npm install` (husky auto-installs no `prepare`).
- O que faz: previne L-021.
- Como debugar: roda manualmente via `npm run pre-commit`.
- Como skip em emergência: `git commit --no-verify` (não recomendado).

## Out of scope

- Hooks de outros tipos (post-commit, pre-push). Foco em pre-commit
  pra L-021.
- Auto-bake-on-yaml-change watcher. Decidido contra (pre-commit é o
  ponto certo de enforcement; watcher complica dev workflow).
- CI check duplicando isso. Pode ser AL-N futuro.
- Lint de YAML files (esquema valida YAML?). AL-25 (a planejar)
  cobre validação contra openapi schema.

## Notas

- husky v9+ é a versão atual estável. Setup mínimo via `npm run prepare`
  (que husky auto-fires em `npm install`).
- Se husky não rodar (ex: dev sem npm install), o hook fica inerte —
  graceful degradation. Audit-via-CI captura como backup.
