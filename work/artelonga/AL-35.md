GIVEN `/analytics/` page existe e faz fetch a
`https://co.artelonga.com.br/api/v1/analytics/public/{summary,recent}`,
AND endpoints retornam 404 hoje, AND fallback localStorage é mostrado
("Fila local · N eventos neste navegador"), AND CO-179 vai criar
esses endpoints,

WHEN CO-179 deploya, AND `/analytics/index.html` opcionalmente atualiza
URLs (se backend usar prefix `/{universe_key}/` em vez de raw `/summary`),

THEN dashboard renderiza dados reais — KPIs, timeseries, top pages,
geo, recent events. Fallback localStorage continua como safety net.

## Blocked by

**CO-179** — public summary/recent endpoints precisam existir.

## Implementação (quando desbloquear)

Verificar shape do response vs o que `/analytics/index.html` consome.
Provavelmente zero mudança no frontend se CO-179 entregar o shape
documentado em AL-22 / CO-179 spec.

Se CO-179 mudar pra path `/{universe_key}/summary`, atualizar URL no
JS de `/analytics/`.

Smoke test: navegar `/analytics`, verificar:
- KPIs preenchidos (não traços)
- Timeseries SVG renderizado
- Top pages tabela com data real
- Geo seção populada (depende de CO-178 deployed)
- Recent events tail
