GIVEN site é pt-BR only, AND certos parceiros/clientes são
internacionais (Yuri tradução EN-PT, communities de impacto global),
AND tradução é trabalho real mas a infra precisa existir antes,

WHEN adicionamos i18n minimal: `lang: en` em profile.yaml campos
opcionais (bio_en, role_en), `<html lang>` ativo, navigation switcher,
URL prefix `/en/` opcional,

THEN copy bilíngue progressivo (cada perfil adiciona EN quando quiser),
SEO Google indexa as duas versões.

## Estrutura proposta

```yaml
# yuri/profile.yaml
bioCurta: "Filho de Kiyoshi e Soninha..."
bioCurta_en: "Son of Kiyoshi and Soninha..."
bio: |
  ...portuguese...
bio_en: |
  ...english...
```

Renderer: lê `<html lang>` ou query `?lang=en`, prefere `_en` variant
se existir, fallback pt-BR.

URL: `/yuri/?lang=en` ou `/en/yuri/` (decisão pendente — query string
mais simples, sem mudar shells).

## Critérios de aceitação (quando ativar)

- [ ] Schema opcional `*_en` fields em Person, Community, Service.
- [ ] Renderer escolhe variant baseado em lang preference.
- [ ] Switcher language no header.
- [ ] `<html lang>` dinâmico.
- [ ] hreflang meta tags pra SEO.
- [ ] Manual smoke test: yuri profile renderiza EN se ?lang=en.

## Out of scope

- Traduções automatizadas (DeepL/Google Translate API).
- Pluralization/gender. PT-BR + EN simples cobrem 95%.
- RTL languages.
- Backlog status — não ativa até demanda concreta.
