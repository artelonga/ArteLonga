GIVEN AL-22 design definiu bake-time popularity strategy (não runtime
fetch — site estático fica robusto), AND CO-180 cria endpoint
`/api/v1/analytics/public/popularity?prefix=/servicos/&days=30`,

WHEN adicionamos `.github/workflows/bake-popularity.yml` que roda
nightly + workflow_dispatch, faz curl no endpoint, escreve
`assets/popularity.json` ordenado deterministicamente, commita,

THEN o JSON fica disponível pro renderer (AL-34) consumir sem
runtime fetch ao backend.

## Blocked by

**CO-180** — endpoint popularity ainda não existe. Esperar mergear.

## Implementação (quando desbloquear)

```yaml
# .github/workflows/bake-popularity.yml
name: bake-popularity
on:
  schedule: [{ cron: '0 4 * * *' }]
  workflow_dispatch: {}
permissions: { contents: write }
jobs:
  bake:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: |
          curl -fsSL "https://co.artelonga.com.br/api/v1/analytics/public/popularity?prefix=/servicos/&days=30" \
            | jq -S '.' > assets/popularity.json
      - uses: stefanzweifel/git-auto-commit-action@v5
        with:
          commit_message: "chore: bake popularity.json"
          file_pattern: assets/popularity.json
```
