GIVEN AL-22 estabelece TypeScript foundation + OpenAPI spec + design
palette em `/design/`, AND `assets/renderer.js` é um arquivo único de
~1900 linhas com dispatcher por `body[data-page]` que renderiza N
páginas (home, profile, service, parceiros, missoes, recursos,
solucoes, sobre, faca-parte, contato, etc), AND a falta de
modularidade complica testes, reuso, e onboarding,

WHEN extraímos componentes reusáveis pra `src/components/*.ts`
(ServiceCard, ProfileCard, FilterChip, ToggleChip, EmptyState,
SearchInput, LocationInput, Badge, Button), AND cada page renderer
vira um arquivo em `src/pages/{home,profile,service}.ts` que compõe
componentes, AND um `src/dispatcher.ts` substitui o dispatcher em
renderer.js (lê `body[data-page]`, importa lazy a page module
correspondente), AND adicionamos build pipeline (Vite ou esbuild)
gerando `assets/renderer.js` a partir do TS,

THEN cada componente tem types do `src/types.ts`, props validados em
compile time, fácil de testar isoladamente, e a `/design/` palette
de AL-22 pode importar os mesmos componentes pra showcase real (não
mock HTML).

## Implementação faseada

### Phase 1: lib helpers + types

`src/lib/{esc,norm,slugify,anglicism}.ts` extraídos do renderer.js.
Dispatcher fica passthrough (importa renderer.js antigo via dynamic
require). Setup Vite. Type checking funcional.

### Phase 2: componentes simples (sem state)

Badge, Button, EmptyState, FilterChip (presentation only). Cada um:
- TS interface
- Render function `(props): string` (HTML string) ou `(props, parent: HTMLElement): void`
- Test snapshot (vitest opcional, ou só smoke check)
- Showcase em `/design/` substituindo o HTML estático.

### Phase 3: componentes com state

ToggleChip, SearchInput (debounce), LocationInput (dropdown custom).
Pattern: cada um expõe um `mount(parent, props)` que retorna unmount.

### Phase 4: páginas

Reescrever cada `src/pages/<page>.ts` consumindo componentes. Começar
pelas mais simples (sobre, recursos), terminar pela home (mais complexa).

### Phase 5: dispatcher

Substituir o dispatcher inline por `src/dispatcher.ts`:
```ts
const pageMap: Record<string, () => Promise<{ render: () => void }>> = {
  home: () => import("./pages/home.ts"),
  profile: () => import("./pages/profile.ts"),
  // ...
};
const page = document.body.dataset.page;
const fn = page ? pageMap[page] : null;
if (fn) {
  try { (await fn()).render(); }
  catch (e) {
    console.error("render falhou:", e);
    document.body.innerHTML = `<main class="main"><p>Algo quebrou. <a href="/">voltar</a></p></main>`;
  }
}
```

Inclui try/catch (L-002) e readyState check (L-001) já estabelecidos.

### Phase 6: cleanup

`assets/renderer.js` agora é só `import "./dispatcher.ts"` ou bundled
output. data.js continua igual (AL-1/2 já migraram people/communities).

## Risk

Migração tão grande tem risco de regressão visual. Mitigações:
- Phase by phase merge (cada phase um PR).
- Visual snapshot tests (Playwright opcional).
- Manual smoke test de cada page principal antes de mergear.
- Rollback fácil: reverter PR mantém site funcional.

## Out of scope

- Server-side rendering / SSR — site continua client-rendered.
- React/Vue/Svelte — vanilla TS. Componentes são funções `(props) =>
  HTMLElement` ou similares, sem framework.
- State management lib — passa props down, callbacks up. Padrão.
- CSS-in-JS — CSS continua nos files existentes (site/components/pages.css).
- Routing lib — sem router; URLs são static + SPA-style render por
  página.
- AL-25 (YAML schema validation no bake) — separado.
