GIVEN o codebase é JS vanilla (renderer.js de ~1900 linhas, data.js ~1500
linhas) sem type safety, sem schema spec single-source-of-truth, sem
componentes reusáveis, AND queremos reduzir bugs de shape (typos em
field names, prop mismatches, etc) que só aparecem em runtime, AND
queremos documentar visualmente o sistema de design pra (a) onboarding
mais rápido, (b) handoff de design futuro, (c) evitar drift visual,

WHEN (a) introduzimos `openapi/artelonga.yaml` como single source of
truth pras shapes (Person, Community, Service, etc) — toda data layer
referencia esse spec; (b) bootstramos TypeScript com `tsconfig.json` +
`src/types.ts` (hand-written espelhando o spec até bumparmos
`openapi-typescript` como devDep); (c) criamos `/design/index.html`
como palette estática que documenta todos os componentes do sistema
em isolamento (cores, tipografia, buttons, badges, chips, cards,
forms, empty states), separando **form** de **content**,

THEN (a) próximas migrations (AL-23 renderer → componentes TS) tem
ground truth pra types, (b) PRs de UI tem referência visual pra
verificar consistência, (c) frontend e backend `co` convergem pro mesmo
schema.

## Princípio: separar form de content

Esse PR institui a regra:
- **Content** vive em high-level abstractions: `<handle>/profile.yaml`,
  `<handle>/community.yaml`, `assets/data.js` (auto-generated), endpoints
  da API `co`. Cada artefato é dado.
- **Form** vive em components reusáveis: CSS classes em
  `assets/{site,components,pages}.css`, futuros TS components em
  `src/components/`. Cada artefato é apresentação genérica.
- **`/design/`** mostra só FORM — sem content específico (sem nomes
  reais de parceiros, sem serviços reais). É o template visual.

Esse princípio está em prática desde AL-1/AL-2 (content em YAML, form
no renderer) — AL-22 só formaliza com TS + OpenAPI.

## Entregas

### 1. `openapi/artelonga.yaml`

Schema completo das entities (Person, Community, Service, Citacao,
HomeLink, Contacts, Parceria, CnaeEntry, etc). Inclui paths existentes
do `co` (`/entries`, `/reindex`). Versão 3.1.0 da spec.

### 2. `tsconfig.json`

`strict: true`, `noUncheckedIndexedAccess: true`,
`exactOptionalPropertyTypes: true`. Target ES2022, module ESNext,
moduleResolution Bundler (compatível com Vite ou esbuild).

### 3. `src/types.ts`

Hand-written interfaces espelhando os schemas do openapi.yaml. Inclui:
- `Handle`, `EntityType`, `Tag`, `Unit`, `ParceriaTipo`
- `Person`, `Community`, `Service`
- `Citacao`, `HomeLink`, `Contacts`, `Parceria`, `Contribuicao`, `CnaeEntry`
- `UniverseData` — shape do `window.AL` runtime (fields + helpers)
- `declare global { interface Window { AL: UniverseData } }`

Quando bumparmos `openapi-typescript` como devDep, vira auto-gerado
via `npm run gen-types`. Por ora hand-written.

### 4. `/design/index.html`

Single static page imports `/assets/bootstrap.js` (CSS herdada do site).
Sections:
- **Color palette** — 11 swatches (preto, brancos, 6 grays, 1 accent).
- **Typography** — 8 níveis da scale (display → eyebrow), letter-spacing
  proportional ao tamanho.
- **Buttons** — primary + secondary. Canvas + code + API table.
- **Badges** — online (digital service), counter (+N children), muted.
- **Chips** — filter chip with count, toggle chip.
- **Card (service)** — 3 variantes: hours×rate, plans, "Sob consulta".
- **Empty state** — sempre com CTA de recuperação.
- **Form fields** — search input, location input com is-default,
  fieldset (contato pattern).

Cada componente: canvas (visual) + pre.dp-code (markup atual ou
proposed TS prop interface) + dp-api-table (props/types/defaults).

`<meta name="robots" content="noindex, nofollow">` — não é content
público, é template.

### 5. CLAUDE.md update

Section "Design system" apontando pra `/design/` + `openapi/artelonga.yaml`
+ `src/types.ts`. Link de "ver palette" no nav.

## Critérios de aceitação

- [x] `openapi/artelonga.yaml` valida como OpenAPI 3.1 (verificar
      manualmente em https://editor.swagger.io ou via openapi-cli).
- [x] `src/types.ts` espelha schemas do yaml — toda interface tem
      campo correspondente no spec.
- [x] `tsconfig.json` strict + indexed access + optional properties.
- [x] `/design/index.html` renderiza no browser (smoke test:
      `python3 -m http.server 8000` + navegar `/design/`).
- [x] Palette mostra cada categoria (color/type/button/badge/chip/card/
      empty-state/form) com canvas + code + (quando aplicável) api-table.
- [x] CLAUDE.md aponta pros novos artefatos.
- [x] V bumpado em `bootstrap.js`.

## Out of scope (separado em AL-23+)

- **Renderer migration** — `assets/renderer.js` continua JS por
  enquanto. AL-23 vai migrar incremental, page-by-page (home, profile,
  service).
- **Vite/esbuild build pipeline** — sem build step ainda. TS compila
  só pra type checking (`npx tsc --noEmit`). AL-23 traz Vite quando
  começar a precisar.
- **`openapi-typescript` codegen** — types.ts hand-written; quando
  começar a divergir do spec, adicionar codegen via `npm run gen-types`.
- **Component library publishing** — components futuros vivem em
  `src/components/` mas não são publicados como pkg. Internal only.
- **Schema validation no bake** — `tools/bake-people.mjs` ainda não
  valida YAML contra openapi schema. AL-25 cobre.
