    Unidade econômica de IA & IaaS — quanto custa servir inteligência

    O que este documento é. Um brief quantitativo sobre custo unitário — por 1M tokens, por requisição, por universo/mês, por GB. Sustenta a tese de monetização do CO: cobra-se o serviço limitado (chamadas de API, storage, universos, capacidade de cómputo), nunca a obra criativa do usuário (“co é livre” — o cérebro é grátis; o que se vende é inteligência limitada como serviço, ver IaaS philosophy).

    Honestidade epistêmica. Cada linha leva selo: [SOURCED] = preço de tabela/fonte citada (jun 2026); [MODELED] = derivado por nós a partir de fontes (premissas explícitas no fim). Isto é em parte um modelo — números de “custo-para-servir por universo” e o crossover local-vs-cloud são construídos, não medidos em produção. A cifra refutada “3.500/me^slocalvs15k cloud” não é usada (ver 1.3 e Notas) — o crossover abaixo é remontado de preços reais de GPU-hora + token.

    Conversão: US1≈R 5,4 (jun 2026, indicativo). Para BRL multiplique os valores acima.

    1. Custo por unidade de serviço de IA

    • Cloud API por 1M tokens — [SOURCED]. Camada de fronteira: Opus 4.8 US5/US 25, GPT-4o US2,5/US 10. Camada barata: Haiku 4.5 US1/US 5, GPT-4.1 nano US0,1/US 0,4 — ~50× mais barata que a fronteira no input. Output custa 4–5× o input em todas as famílias; logo o custo unitário real depende da razão out:in da tarefa (resumir = barato; gerar = caro). Maritaca/Sabiá-3 cobra em BRL e ~3–4× mais barato/token que GPT-4o (ver 1.3) — relevante para residência de dados no Brasil.
    • Dois multiplicadores que mudam tudo — [SOURCED]. Batch −50% (tarefas não interativas). Prompt caching: leitura ~0,1×, escrita ~1,25× — num agente com prefixo grande estável (system + ferramentas), o custo efetivo do prefixo despenca ~90% após a 1ª chamada. Para um IaaS, cache + batch são as duas alavancas de margem mais fortes sem trocar de modelo.
    • Custo por requisição — [MODELED]. Não há “preço por request”; é (in×preço_in + out×preço_out). Ex.: um Q&A de 2k in / 0,5k out no Haiku ≈ US0,0045∗∗;noOpus≈∗∗US 0,0225 (~5×). Numa requisição típica de chat o output domina o custo.
    • Embeddings — [MODELED]. text-embedding-3-small a **US0,02/1Mtok∗∗(US 0,01 em batch). A ~500 tok/doc, 1k docs ≈ US0,01∗∗;∗∗1Mdocs≈US 10. Embedding é barato; o custo recorrente migra para o vector DB.
    • Vector DB — [SOURCED]. Pinecone serverless ≈ US7/me^s@10Mvetores∗∗( ∗∗US 0,70/1M vetores), **US700+/me^s@100M∗∗;modelowrite/read(US 4/1M writes, US$ 16/1M reads). Aviso de fonte: contas reais vêm 2,5–4× acima da calculadora. Alternativa de menor custo unitário: pgvector/Qdrant auto-hospedado (só o custo de máquina+volume — ver §2).

    3. Crossover local vs cloud — [MODELED] (cifra “$3.500” não usada)

    Remontagem a partir de GPU-hora real + preço-de-token real, não da cifra refutada. Pergunta: a partir de que volume um GPU alugado/possuído bate o pague-por-token?

    Premissas do modelo: requisição média 1.000 tok out; throughput de um GPU médio servindo um modelo open ~30 tok/s sustentado ⇒ ~108k req/dia teóricos (na prática 30–50% disso por overhead/batching).

    Conclusão honesta. O crossover não é um número único — depende de utilização. Regra de bolso [MODELED]: abaixo de ~poucos milhares de req/dia, a API ganha (capex/idle de GPU desperdiça); acima disso, GPU alugado/possuído ganha — desde que a utilização seja alta. Uma GPU ociosa é o pior dos mundos (paga capex, serve pouco). Para o CO/M4 Max: roda 14B (~9 GB) para coexistir com trabalho; 30B/32B só em máquina ociosa (36 GB é apertado → swap → máquina inteira lenta — ver CLAUDE.md). O argumento defensável de “local” não é custo garantido — é residência de dados + ausência de taxa-por-token + privacidade arquitetural (ver §5 e 1.3).

    4. Unidade econômica da plataforma (IaaS como o CO)

    • Custo-para-servir por universo pequeno — [MODELED]: ~US2–8/me^s.∗∗Piso≈maˊquinaFly(US 2) + volume pequeno (US0,15–1)+egressmodesto;sobecomtraˊfegoetamanhodo‘data.db‘.∗∗Scale−to−zero∗∗derrubauniversosraramente−acessadospara∗∗ US 0,15–0,50/mês. O CO já separa storage (CO-77: meta.db global + data.db por universo) — o custo marginal de adicionar um universo é quase só o volume, não uma nova máquina.
    • Custo por usuário ativo — [MODELED]. Como o conteúdo é markdown servido estaticamente (CO-93 public-static), o custo de leitura tende a egress (US$ 0,02/GB) — fração de centavo por visita. O custo que escala com o usuário é inteligência (tokens), não hosting — exatamente o que a tese “cobra-se o serviço, não a obra” isola: storage/serve é commodity barata; inteligência limitada é o produto medido.
    • Economia de crescimento de storage — [SOURCED]+[MODELED]. Volume linear a US0,15/GB/me^s—∗∗previsıˊvel∗∗,mas∗∗cobraparado∗∗(lic\c​a~odoincidentededisco−cheio).SemcamadaS3,amatemaˊticaderetenc\c​a~odesnapshotsquebra(volumede3GB;vermemoˊriadoprojeto).∗∗Proˊximopassodemargem=StaaS/S3∗∗(objeto US 0,02/GB/mês, ~7× mais barato que volume block, e elástico).
    • Estrutura de margem do “preço por capacidade” — [MODELED]. Vender capacidade de serviço (cotas de tokens/cómputo/storage) tem margem alta porque (a) o cérebro/leitura é grátis/commodity, (b) cache+batch cortam COGS ~50–90%, (c) o custo marginal do conteúdo é ~egress. O risco de margem é inteligência de fronteira repassada a preço-tabela — daí a importância de rotear por tier (Haiku/Sabiá para o comum, Opus só quando preciso) e de modelos locais para zerar a taxa-por-token onde a qualidade 14B basta.
    • Comparáveis open/self-host (per-seat vs uso) — [SOURCED]. SaaS per-seat (ex.: vector DB Pinecone Builder US20/me^sflat,StandardUS 50 mín.) descola custo de uso — bom para o fornecedor, ruim para o usuário esporádico. Uso puro (token/GB) alinha preço a consumo mas é imprevisível (contas de vector DB 2,5–4× acima do estimado). O CO fica do lado uso + cota com piso de hosting commodity — o oposto do lock-in per-seat.

    Leitura unitária. As defesas de privacidade que o CO adota são quase grátis em custo unitário — DP ~0, C2PA desprezível, local zera taxa-por-token (troca por capex/latência). FHE é o único caro e por isso fica reservado a lotes estreitos offline de alta sensibilidade, nunca caminho default (espelha o P9 honesto de 1.3). Ou seja: privacidade-por-desenho não é um imposto sobre a margem — é compatível com a estrutura de custo do IaaS.
